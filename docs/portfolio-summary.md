## 3. 기획 배경 및 문제 정의

### 문제 인식
| 기존 문제 | 현장 상황 |
|-----------|-----------|
| 외부 AI(ChatGPT 등)에 사내 데이터 입력 불가 | 제조업 보안 규정상 외부 전송 금지 |
| SQL 모르면 DB 조회 불가 | 현업이 매번 IT팀에 데이터 요청 → 병목 |
| 사내 문서(Confluence 등) 분산 | 신규 입사자가 정보 찾기 어려움 |
| 소스코드 이해에 시간 소요 | 인수인계 비용 증가 |

### 해결 방향
- 폐쇄망에서도 동작하는 자체 AI 플랫폼 구축
- 자연어 → SQL 자동 변환으로 비개발자도 데이터 접근 가능
- RAG(검색 증강 생성) 기반 사내 문서 AI 검색
- Git 저장소 색인으로 코드 기반 AI 분석

## 4. 시스템 아키텍처

```
사용자 (자연어 질문)
    │
    ▼
┌─────────────────────────────────────────┐
│        Enterprise LLM Platform           │
│                                          │
│  [Text-to-SQL]  [RAG 문서검색]  [Git RAG] │
│       │              │             │      │
│       └──────┬───────┘             │      │
│              ▼                     │      │
│      [통합 진단 Analyze]  ←────────┘      │
│              │                            │
│              ▼                            │
│     LLM (GPT / 자체모델 전환 가능)         │
└─────────────────────────────────────────┘
       │            │            │
  Oracle DB     ChromaDB      Git Repo
 (실제 업무DB)  (벡터 DB)    (소스코드)
```

### 핵심 설계 원칙
1. **Deterministic Backbone**: AI는 판단·생성에만 사용, 검증·실행은 코드로 처리
2. **이중 모드**: `MODE=local` (OpenAI API) / `MODE=airgap` (자체 모델) 전환
3. **모듈 독립성**: Text-to-SQL, RAG, Git RAG가 독립적으로 동작, 통합 진단에서 조합

## 5. 핵심 기능 상세

### 5-1. Text-to-SQL (자연어 → DB 조회)
- **원리**: DB 스키마(테이블 구조)를 텍스트로 변환 → LLM 프롬프트에 첨부 → LLM이 SQL 생성 → 백엔드가 DB에 실행
- **지원 DB**: Oracle, PostgreSQL, MySQL
- **안전장치**: SELECT만 허용 (DELETE/DROP/UPDATE 차단)
- **실증 결과**: "A라인 이번달 불량률 알려줘" → Oracle 실시간 조회 → "2.47%"

### 5-2. RAG (문서 기반 AI 질의응답)
- **문서 수집**: 파일 업로드 + Confluence 자동 동기화
- **검색 파이프라인**: Semantic Chunking → 임베딩 → BM25 하이브리드 검색 → Reranker
- **문서 뷰어**: 검색 결과에서 원본 문서 바로 열람, Confluence 원본 링크 연결
- **실증 결과**: "품질 검사 기준 알려줘" → Confluence 문서에서 관련 내용 추출 → 답변

### 5-3. Git Code RAG (소스코드 AI 분석)
- Git 저장소(로컬/원격)를 벡터 DB에 색인
- "인증 로직 어디서 처리돼?" → 관련 코드 위치 + 설명

### 5-4. 통합 진단 (Analyze)
- RAG(문서) + Text-to-SQL(DB 데이터) 동시 실행 → 종합 답변
- "A라인 불량 급증 원인 분석해줘" → DB에서 수치 조회 + 문서에서 규정 검색 → 종합

### 5-5. 폐쇄망 대응 (Airgap Mode)
- 외부 인터넷 없이 자체 LLM 모델로 동작 가능
- 임베딩 모델도 로컬 저장 → 완전 오프라인 동작
- 보안이 중요한 제조업·금융·공공기관 환경 대응

## 11. 면접 예상 질문 대비

**Q: Text-to-SQL이 뭔가요?**
A: DB 테이블 구조(스키마)를 텍스트로 변환해서 LLM에게 전달하면, LLM이 질문에 맞는 SQL을 생성합니다. 우리 백엔드가 그 SQL을 실제 DB에 실행하고 결과를 다시 LLM에게 보내서 자연어 답변을 만듭니다.

**Q: RAG가 뭔가요?**
A: LLM이 학습하지 않은 사내 문서를 검색해서 프롬프트에 첨부하는 기술입니다. 문서를 벡터(숫자 배열)로 변환해서 저장해두고, 질문이 들어오면 유사한 문서를 찾아서 LLM에게 "이 문서를 참고해서 답해"라고 전달합니다.

**Q: 폐쇄망에서 어떻게 AI를 쓰나요?**
A: 두 가지 모드를 설계했습니다. 인터넷 가능 환경에서는 OpenAI API를 쓰고, 폐쇄망에서는 자체 LLM 모델(GPT-OSS 등)을 로컬에 배포합니다. 임베딩 모델도 로컬에 저장해서 완전히 외부 연결 없이 동작합니다.

**Q: 왜 직접 만들었나요? 기존 솔루션을 도입하면 안 되나요?**
A: 기존 솔루션(LangChain 등)은 범용적이라 특정 업무에 맞추기 어렵고, 폐쇄망 대응이 부족합니다. 직접 구축하면 제조업 도메인에 최적화할 수 있고, 각 모듈을 독립적으로 교체·확장할 수 있습니다. 또한 기술 원리를 깊이 이해해야 올바른 기획을 할 수 있다고 판단했습니다.

**Q: 이 경험을 ICT 기획에 어떻게 활용하겠나요?**
A: AI 기술의 가능성과 한계를 모두 아는 기획자가 될 수 있습니다. "이건 AI로 가능하다/불가능하다"를 직접 판단할 수 있어서, 현실적인 기획과 정확한 요구사항 정의가 가능합니다. 또한 제조업 현장 경험이 있어서 현업의 needs를 기술 솔루션으로 연결하는 데 강점이 있습니다.
